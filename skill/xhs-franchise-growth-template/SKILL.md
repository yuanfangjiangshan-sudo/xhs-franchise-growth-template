# Xiaohongshu Franchise Growth Template

## Core Positioning

Use this skill as a reusable workflow for Xiaohongshu accounts that need trust-building content plus light conversion.

Before first use, read and adapt:

- `references/core-knowledge.md`
- `references/account-grounding.md`
- `references/source-map-template.md`

Default stance:

- Build trust before conversion.
- Make each note useful even if the reader never buys.
- Prefer concrete business problems over generic inspiration.
- Keep claims grounded in available source material.
- Avoid hard-selling unless the user explicitly asks.

## Required Workflow

### 1. Account Diagnosis

Read `references/account-grounding.md`.

Identify:

- latest published-note status
- recent exposure, engagement, saves, comments, or follower movement
- strongest recent pattern
- weakest signal
- data limitations

If no account data is available, say so and generate only a strategy sample or draft sample.

### 2. Benchmark Research

Read `references/benchmark-research.md`.

Inspect recent same-topic or same-audience Xiaohongshu examples when available. Focus on:

- topic angle
- title pattern
- cover hook
- page structure
- comment trigger
- audience fit

Borrow patterns only. Do not copy text, identity, images, or proprietary claims.

If benchmark pages require login or authorization, ask the user to log in or provide examples.

### 3. Source Grounding

Read `references/core-knowledge.md` first.

When the topic needs brand, product, offer, service, or proof details, read `references/source-map-template.md` and use only approved materials.

If a number, promise, customer story, or operating detail is uncertain, avoid it or ask the user for confirmation.

### 4. Draft Generation

Read `references/output-template.md`.

Shape the note around one concrete reader pain, such as:

- lead generation
- enrollment conversion
- parent communication
- operational efficiency
- team training
- product differentiation
- local-market competition
- AI-enabled workflow improvement

Use brand or offer details as supporting proof, not as the whole note.

### 5. Risk Check

Before finalizing, check for:

- guaranteed returns
- exaggerated income claims
- zero-risk promises
- unauthorized customer names
- private data
- legal or policy claims stated too definitively
- platform-rule violations

Use cautious language when local policy, franchise rules, advertising rules, or education regulations may matter.

## Output Contract

For daily content generation, output these sections:

1. 生成前依据
2. 账号诊断结论
3. 对标/热门内容观察
4. 今日选题
5. 选题角度说明
6. 标题备选
7. 小红书文-图配套文案
8. 小红书发布正文
9. 标签建议
10. 评论关键词设计
11. 私信承接话术
12. 风险检查
13. 发布建议

Follow the detailed format in `references/output-template.md`.

## Review And Iteration

When the user asks for weekly review, read `references/review-loop.md`.

When performance data is provided:

- summarize what worked
- summarize what failed
- identify reusable title, cover, topic, and conversion-script rules
- propose the smallest useful update to `references/core-knowledge.md` or another reference file

Only edit skill files when the user explicitly asks to update the workflow.
